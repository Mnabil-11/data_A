# SQL Sales Analytics — Superstore

A SQL + Power BI analytics project covering KPI discovery, root-cause "why" analysis,
window-function time series, and RFM marketing segmentation on the Superstore sales
dataset (9,800 order line items / 4,922 orders / 793 customers, 2015–2018).

## Data reality check (read this first)

This dataset's `train.csv` contains **Sales only** — there is no Profit, Discount, or
Quantity column. Every query, KPI, and insight in this project is therefore built on
Sales (revenue) as the sole performance metric. Anywhere a classic Superstore analysis
would use Profit or Discount (e.g. "most profitable category," "avg discount by region"),
this project re-scopes it to the nearest honest Sales-based equivalent and calls that out
explicitly in the query comments and in [REPORT.md](REPORT.md).

## How to reproduce

1. **Load the data into your own engine** (PostgreSQL/SQL Server) using
   `sql/01_database_setup.sql`, then bulk-load `data/train.csv` per the load command
   in that file's comments. All other scripts are written to run unmodified on either
   engine — dialect differences (e.g. `DATEDIFF` vs. date subtraction, `TOP` vs.
   `LIMIT`) are called out inline.
2. **Or skip straight to querying:** open `database/superstore.duckdb` (e.g. with the
   [DuckDB CLI](https://duckdb.org) or DBeaver) — it already has the table, data, and
   indexes from step 1 applied, so you can run any file in `sql/` immediately.
3. **Run the scripts in order** (`01` → `07`) — each later script assumes the schema
   from `01_database_setup.sql` and, where relevant, reuses logic introduced earlier
   (e.g. the RFM CASE rules referenced again in `07_business_insights.sql`).
4. **Build the dashboard** — see [dashboard/README.md](dashboard/README.md).

## Verification note

Every query in `sql/` was executed against the real 9,800-row dataset (via DuckDB)
before being committed here — none of it is theoretical or untested. One real bug was
caught and fixed in the process: `NTILE()` in the RFM scoring produced different segment
counts on different runs because ties in `frequency`/`recency`/`monetary` have no
guaranteed order without an explicit tiebreaker — `customer_id` was added as a
deterministic tiebreaker in every `NTILE(...) OVER (ORDER BY ...)` call to fix it (see
the comment in `sql/05_rfm_analysis.sql`).
